## 왜 VPC가 필요한가

AWS는 수백만 고객이 같은 물리 서버를 공유한다. VPC(Virtual Private Cloud)는 AWS 위에 나만의 가상 네트워크를 만드는 것으로, 다른 고객과 완전히 격리된 환경을 제공한다.

## 구성 요소

### CIDR 블록

```
VPC CIDR: 10.0.0.0/16
→ 10.0.0.0 ~ 10.0.255.255 (약 65,000개 IP)
```

### 서브넷 - 목적별 분리

```
VPC (10.0.0.0/16)
├── 퍼블릭 서브넷  → 인터넷 통신 가능 (웹 서버, ALB)
├── 프라이빗 서브넷 → 인터넷 직접 접근 불가 (앱 서버)
└── 프라이빗 서브넷 → RDS
```

### 인터넷 게이트웨이 (IGW)

VPC와 인터넷 사이의 출입문. 퍼블릭 서브넷의 라우팅 테이블에만 IGW 경로 존재.

### NAT 게이트웨이

프라이빗 서브넷 → 인터넷 **단방향** 출구. 외부에서 먼저 들어오는 건 차단.

```
[프라이빗 EC2] → [NAT GW (퍼블릭)] → [IGW] → [인터넷]
외부 → 내부: 차단
```

### 보안 그룹 vs 네트워크 ACL

| 구분 | 보안 그룹 | 네트워크 ACL |
| --- | --- | --- |
| 적용 단위 | EC2 인스턴스 | 서브넷 전체 |
| 상태 | Stateful (응답 자동 허용) | Stateless (인/아웃 따로 설정) |
| 기본값 | 모든 인바운드 차단 | 모든 트래픽 허용 |
| 규칙 | 허용만 | 허용/차단 모두 |

## 실제 설계 패턴: 3-Tier 아키텍처

```
인터넷
  ↓
[ALB] - 퍼블릭 서브넷 (AZ-A, AZ-B)
  ↓
[EC2 앱 서버] - 프라이빗 서브넷 (AZ-A, AZ-B)
  ↓
[RDS] - 프라이빗 서브넷 (AZ-A, AZ-B)
```

모든 레이어를 AZ 두 개에 배치 → AZ 하나 장애나도 서비스 유지 = **고가용성(HA)**

## VPC 피어링

두 VPC를 프라이빗으로 연결. 단, **전이적 라우팅 불가** (A↔︎B↔︎C여도 A→C 직접 통신 불가).
