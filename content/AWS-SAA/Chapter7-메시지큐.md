## 디커플링이 왜 필요한가

직접 연결 문제: 하나 다운 → 연쇄 장애 / 트래픽 급증 → 동시 과부하 / 새 서비스 → 코드 수정
메시지 큐 도입: 각 서비스 독립, 큐가 완충 역할

## SQS

**동작 방식:** 생산자 → 저장 → 소비자가 꺼냄 (Pull) → 처리 완료 → 삭제

**Visibility Timeout:** 처리 중 다른 소비자는 못 봄. 실패 시 재처리 가능.

**DLQ:** n번 실패 → 별도 큐로 이동 → 분석/재처리.

### 표준 vs FIFO

| 구분 | 표준 | FIFO |
| --- | --- | --- |
| 순서 보장 | 없음 | 정확한 순서 |
| 중복 | 가끔 발생 | 정확히 1번만 |
| 처리량 | 무제한 | 초당 최대 3,000건 |
| 용도 | 순서 무관 | 주문, 결제 |

## SNS

1개 메시지 → 구독한 모든 대상에게 동시 전달 (Push).

**팬아웃 패턴:**

```
[이벤트] → [SNS] → [SQS A] → 소비자 1
                 → [SQS B] → 소비자 2
                 → [SQS C] → 소비자 3
```

## EventBridge

AWS 서비스 이벤트 + SaaS 이벤트를 조건 기반으로 라우팅하는 허브.

## 선택 기준

| 상황 | 서비스 |
| --- | --- |
| 메시지 보관, 1:1 처리 | SQS |
| 순서 보장 | SQS FIFO |
| 여러 소비자 동시 전달 | SNS + SQS 팬아웃 |
| AWS 서비스 이벤트, 조건부 라우팅 | EventBridge |
