# 오픈소스 쿠버네티스 아키텍처 설계 및 구현

**학습 기간:** 2026.02.23 ~ 2026.03.04

---

### 1. Kubernetes 개요

**왜 Kubernetes인가?**

Docker만으로는 대규모 운영에 한계가 있음.

| 문제 | Docker 한계 |
| --- | --- |
| 컨테이너 죽으면? | 자동 복구 없음 |
| 여러 서버 배포? | 서버마다 수동 SSH |
| 로드밸런싱? | 별도 설정 필요 |
| 선언적 관리? | 불가 |

**Kubernetes의 핵심 철학 — 선언적(Declarative) 모델**

```yaml
spec:
  replicas: 3  # "nginx는 항상 3개 유지"
```

선언만 하면 K8s가 알아서 현재 상태를 원하는 상태로 맞춤. 죽으면 재생성, 노드 장애 시 다른 노드로 재배치.

**Docker vs Kubernetes**

| 구분 | Docker | Kubernetes |
| --- | --- | --- |
| 목적 | 컨테이너 실행 | 컨테이너 오케스트레이션 |
| 관리 단위 | 컨테이너 | Pod |
| 자동 복구 | 없음 | 있음 |
| 스케일링 | 수동 | 자동(HPA) |
| 로드밸런싱 | 별도 설정 | Service |

### 2. 아키텍처

```
         ┌────────────────────┐
         │   Control Plane    │
         │  API Server        │
         │  Scheduler         │
         │  Controller Mgr    │
         │  etcd              │
         └──────────┬─────────┘
                    │
   ┌────────────────┼────────────────┐
   │                │                │
┌──▼──────┐   ┌──▼──────┐   ┌──▼──────┐
│ Worker1 │   │ Worker2 │   │ Worker3 │
│ kubelet │   │ kubelet │   │ kubelet │
│ kube-   │   │ kube-   │   │ kube-   │
│  proxy  │   │  proxy  │   │  proxy  │
│containerd│  │containerd│  │containerd│
└─────────┘   └─────────┘   └─────────┘
```

**Control Plane 구성요소**

| 구성요소 | 역할 |
| --- | --- |
| API Server | 모든 요청의 진입점, kubectl 명령의 목적지 |
| etcd | 분산 KV 저장소, 클러스터 전체 상태 저장 (K8s의 DB) |
| Scheduler | 새 Pod을 어느 노드에 배치할지 결정 |
| Controller Manager | 현재 상태 ↔ 원하는 상태 비교 후 수정 (Reconciliation Loop) |

**Worker Node 구성요소**

| 구성요소 | 역할 |
| --- | --- |
| kubelet | API Server 지시 수신, Pod 생성/상태 모니터링 |
| kube-proxy | Service 네트워크 구현, Pod 간 트래픽 라우팅 |
| containerd | 실제 컨테이너 실행 (현재 표준 런타임) |

**Pod 생성 흐름**

```
kubectl apply → API Server → etcd 저장 → Controller 감지
→ Scheduler 노드 선택 → kubelet Pod 생성 → containerd 컨테이너 실행
```

### 3. 핵심 오브젝트

**Namespace** — 논리적 격리 공간

```bash
kubectl create namespace dev
kubectl get pods -n dev
```

| 기본 Namespace | 역할 |
| --- | --- |
| default | 기본 작업 공간 |
| kube-system | 시스템 Pod 실행 |
| kube-public | 공개 리소스 |
| kube-node-lease | 노드 상태 관리 |

**Pod** — 컨테이너가 실행되는 최소 단위

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
spec:
  containers:
  - name: nginx
    image: nginx:latest
    ports:
    - containerPort: 80
```

**Deployment** — Pod을 관리하는 오브젝트 (실무에서 가장 많이 사용)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:latest
```

```bash
kubectl apply -f deploy.yaml
kubectl get deployments
kubectl rollout undo deployment/nginx-deploy   # 롤백
```

**ReplicaSet** — replicas 수 유지 담당. 보통 Deployment가 자동 생성하므로 직접 쓰는 경우 드묾.

### 4. 워크로드 타입

| 타입 | 용도 | 특징 |
| --- | --- | --- |
| Deployment | 무상태 웹/API 서버 | Rolling Update, 롤백 |
| StatefulSet | DB, 메시지큐 | Pod 이름 고정, PV 연결, 순차 배포 |
| DaemonSet | 로그 수집기, 모니터링 에이전트 | 모든 노드에 1개씩 |
| Job | 배치 작업 | 완료 후 종료 |
| CronJob | 정기 실행 작업 | cron 스케줄 기반 |

**StatefulSet 특징**

- Pod 이름 고정: `mysql-0`, `mysql-1`, `mysql-2`
- 순차 생성/삭제 보장
- Pod마다 독립 PVC 연결
- Headless Service 필수 (`clusterIP: None`)

**DaemonSet 특징**

- replicas 없음, 노드 수 = Pod 수
- 노드 추가 시 자동 배포
- 사용 사례: Fluentd(로그), Node Exporter(모니터링)

### 5. Service

Pod IP는 재시작 시마다 변경됨 → Service로 고정 접근점 제공

| 타입 | 외부 접근 | 사용 목적 |
| --- | --- | --- |
| ClusterIP | ❌ | 클러스터 내부 통신 (기본값) |
| NodePort | ✅ (노드IP:포트) | 테스트, 온프레미스 |
| LoadBalancer | ✅ (외부 IP) | 클라우드 운영 환경 |

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-svc
spec:
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
  type: ClusterIP
```

### 6. Ingress

여러 서비스를 단일 진입점으로 라우팅. Ingress Controller(nginx-ingress 등)가 필요.

```
Client → Ingress Controller → Ingress 규칙 → Service → Pod
```

**도메인 기반 / 경로 기반 라우팅**

```yaml
rules:
- host: app1.local
  http:
    paths:
    - path: /
      backend:
        service:
          name: app1-svc
- host: app2.local
  http:
    paths:
    - path: /
      backend:
        service:
          name: app2-svc
```

온프레미스에서 LoadBalancer IP 할당은 **MetalLB** 사용.

### 7. PV / PVC / StorageClass

Pod 삭제 시 내부 데이터 소멸 → 영구 저장소 필요

| 개념 | 설명 |
| --- | --- |
| PV (PersistentVolume) | 관리자가 미리 만들어둔 저장소 자원 |
| PVC (PersistentVolumeClaim) | 사용자가 저장소를 요청하는 객체 |
| StorageClass | 동적 프로비저닝 정책 템플릿 |

**정적 프로비저닝:** 관리자가 PV 생성 → PVC 생성 → 자동 바인딩

**동적 프로비저닝:** StorageClass 설정 → PVC 생성 → Provisioner가 PV 자동 생성

**AccessMode**

- `ReadWriteOnce`: 단일 노드에서 읽기/쓰기
- `ReadOnlyMany`: 여러 노드에서 읽기
- `ReadWriteMany`: 여러 노드에서 읽기/쓰기

### 8. ConfigMap / Secret

코드와 설정 분리. 재배포 없이 설정 변경 가능.

| 항목 | ConfigMap | Secret |
| --- | --- | --- |
| 목적 | 일반 설정 | 민감한 데이터 |
| 데이터 형식 | 평문 | Base64 인코딩 |
| 사용 예 | 로그 레벨, 포트 | 비밀번호, API 키 |

```yaml
# ConfigMap
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: "info"
  DATABASE_URL: "postgres://db:5432"
```

```yaml
# Secret
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
stringData:
  password: "mysecret"
```

### 9. RBAC / ServiceAccount

**API 요청 처리 순서:** Authentication(인증) → Authorization(인가) → Admission Control

**RBAC 구성요소**

| 오브젝트 | 범위 | 설명 |
| --- | --- | --- |
| Role | Namespace | 특정 Namespace 내 권한 정의 |
| ClusterRole | 클러스터 전체 | 클러스터 범위 권한 정의 |
| RoleBinding | Namespace | Role을 사용자/SA에 연결 |
| ClusterRoleBinding | 클러스터 전체 | ClusterRole을 사용자/SA에 연결 |

### 10. Helm

K8s 패키지 매니저. 여러 YAML 파일을 하나의 Chart로 패키징하여 관리.

**핵심 용어**

| 용어 | 설명 |
| --- | --- |
| Chart | K8s 매니페스트 템플릿 묶음 |
| Release | Chart의 실제 배포 인스턴스 |
| Repository | Chart 저장소 (apt와 동일 개념) |
| values.yaml | 템플릿에 주입할 변수 기본값 |

```bash
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
helm install my-nginx bitnami/nginx        # 설치
helm upgrade my-nginx bitnami/nginx        # 업그레이드
helm rollback my-nginx 1                   # 롤백
helm uninstall my-nginx                    # 삭제
helm list                                  # 배포 목록
```

### 11. Resource 관리 (Requests / Limits / HPA)

**Requests vs Limits**

| 항목 | 설명 |
| --- | --- |
| requests | 스케줄링 기준 (이 노드에 배치 가능한지 판단) |
| limits | 실행 중 최대 자원 상한 (초과 시 CPU throttling / Memory OOMKilled) |

**QoS 클래스** (퇴출 우선순위: BestEffort → Burstable → Guaranteed)

| 클래스 | 조건 |
| --- | --- |
| Guaranteed | requests == limits |
| Burstable | requests < limits |
| BestEffort | requests/limits 모두 미지정 |

**HPA (Horizontal Pod Autoscaler)**

```yaml
# CPU 70% 초과 시 자동 확장
metrics:
- type: Resource
  resource:
    name: cpu
    target:
      type: Utilization
      averageUtilization: 70
minReplicas: 2
maxReplicas: 10
```

### 12. Probe (헬스체크)

Pod이 Running이어도 서비스 정상 보장 안 됨 → Probe로 자동 감지/복구

| 구분 | Liveness Probe | Readiness Probe |
| --- | --- | --- |
| 질문 | "컨테이너가 살아있나?" | "지금 트래픽 받아도 되나?" |
| 실패 시 | 컨테이너 재시작 | Service Endpoint에서 제외 |
| 목적 | Deadlock 복구 | 초기화 전 트래픽 차단 |

**Probe 방식:** httpGet / exec / tcpSocket

**공통 옵션**

| 옵션 | 의미 |
| --- | --- |
| initialDelaySeconds | 첫 검사까지 대기 시간 |
| periodSeconds | 검사 주기 |
| failureThreshold | 연속 실패 횟수 기준 |
