### 1. DNS (Domain Name System)

**정의**
도메인 이름을 IP 주소로 변환하는 계층적 분산 데이터베이스 시스템.

```
www.example.com → 192.168.80.100
```

**DNS 계층 구조**

```
.              ← Root (ICANN 관리)
└── com        ← TLD (등록기관 관리)
    └── example  ← 2차 도메인 (사용자 임대)
        └── www  ← 호스트 이름
```

도메인은 "구매"가 아닌 **"임대"** 개념 (보통 1년 단위 갱신)

**DNS 서버 종류**

| 종류 | 역할 |
| --- | --- |
| Caching Name Server (Recursive) | 재귀 질의 수행, 결과를 TTL 동안 캐시 저장 |
| Authoritative Name Server | 특정 Zone의 원본 데이터 보유, 정답 제공 |

**DNS 질의 흐름**

```
클라이언트 → Caching DNS → Root → TLD → Authoritative DNS
                 ↑
            TTL 동안 캐시 저장 (두 번째 요청은 캐시 응답)
```

**주요 레코드 타입**

| 레코드 | 역할 |
| --- | --- |
| A | 도메인 → IPv4 주소 |
| AAAA | 도메인 → IPv6 주소 |
| PTR | IP → 도메인 (역방향) |
| CNAME | 별칭 (Alias) |
| NS | 권한 서버 지정 |
| MX | 메일 서버 지정 |
| SOA | Zone 기본 정보 |

**DNS 사용 포트**

| 프로토콜 | 포트 | 용도 |
| --- | --- | --- |
| UDP | 53 | 일반 DNS 질의 |
| TCP | 53 | Zone Transfer, 큰 응답 |

**BIND9 설정 (Ubuntu)**

```bash
sudo apt install -y bind9 bind9-utils bind9-dnsutils

# 설정 파일 위치
/etc/bind/named.conf          # 메인 설정
/etc/bind/named.conf.options  # 서버 옵션
/etc/bind/named.conf.local    # 사용자 정의 Zone

# Zone 파일 위치
/etc/bind/  또는  /var/named/
```

**Zone File 기본 구조**

```
$TTL 86400
@   IN  SOA ns1.example.com. admin.example.com. (
        2026022101  ; Serial
        3600        ; Refresh
        900         ; Retry
        604800      ; Expire
        86400       ; Minimum TTL
)
    IN  NS  ns1.example.com.
ns1 IN  A   192.168.80.53
www IN  A   192.168.80.100
```

**Zone Transfer (Master/Slave 이중화)**

- AXFR: 전체 전송
- IXFR: 증분 전송
- Slave는 SOA Serial 비교 후 동기화
- TCP 53 사용

**named-chroot 보안**
DNS 서버 프로세스의 루트 디렉터리를 `/var/named/chroot`로 제한. 침해 발생 시 피해 범위 최소화 (보안 격리).

### 2. VPN (Virtual Private Network)

**정의**
공중 네트워크(인터넷)를 통해 사설 네트워크처럼 안전하게 통신하는 기술. 터널링으로 데이터를 암호화하여 전송.

**VPN 종류**

| 종류 | 설명 | 사용 사례 |
| --- | --- | --- |
| Site-to-Site VPN | 두 거점(본사↔지사) 간 연결 | 기업 지점 간 내부망 연결 |
| Remote Access VPN | 개인 사용자 → 기업 내부망 연결 | 재택근무, 원격 접속 |

**터널링 (Tunneling)**
패킷을 다른 프로토콜로 캡슐화하여 전송. 외부에서는 내용을 알 수 없음.

**IPsec**
네트워크 계층(L3)에서 IP 패킷을 암호화/인증하는 프로토콜 스위트.

| 구성요소 | 역할 |
| --- | --- |
| AH (Authentication Header) | 인증만, 암호화 없음 |
| ESP (Encapsulating Security Payload) | 인증 + 암호화 |
| IKE (Internet Key Exchange) | 키 교환 및 SA 협상 |

**IKE 2단계**

- Phase 1: IKE SA 협상 (서로 인증, 암호화 방식 합의)
- Phase 2: IPsec SA 협상 (실제 데이터 전송용 터널 설정)

**IPsec 모드**

| 모드 | 설명 |
| --- | --- |
| Transport Mode | 페이로드만 암호화, 원본 IP 헤더 유지 |
| Tunnel Mode | 전체 패킷 암호화 + 새 IP 헤더 추가 (Site-to-Site에 사용) |

**GRE (Generic Routing Encapsulation)**
IPsec과 함께 사용되는 터널링 프로토콜. 멀티캐스트/브로드캐스트 지원 (IPsec 단독으로는 불가).

### 3. BGP (Border Gateway Protocol)

**정의**
서로 다른 AS(Autonomous System) 간 라우팅 정보를 교환하는 **EGP(외부 게이트웨이 프로토콜)**. 인터넷 백본 라우팅에 사용.

**AS (Autonomous System)**
단일 관리 주체 하에 운영되는 IP 네트워크 집합. AS 번호(ASN)로 식별.

| ASN 범위 | 설명 |
| --- | --- |
| 1 ~ 64511 | 공인 ASN (IANA 할당) |
| 64512 ~ 65535 | 사설 ASN (내부용) |

**BGP 특징**

| 항목 | 내용 |
| --- | --- |
| 프로토콜 타입 | EGP (External Gateway Protocol) |
| 전송 프로토콜 | TCP 179 |
| 라우팅 기준 | AS Path, 정책(Policy) 기반 |
| 수렴 속도 | 느림 (대신 안정성 높음) |
| 주요 용도 | ISP 간 연결, 인터넷 백본 |

**BGP Neighbor 연결 과정 (피어링)**

```
Idle → Connect → Active → OpenSent → OpenConfirm → Established
```

Established 상태가 되어야 라우팅 정보 교환 가능.

**iBGP vs eBGP**

| 구분 | iBGP | eBGP |
| --- | --- | --- |
| 적용 범위 | 동일 AS 내부 | 서로 다른 AS 간 |
| TTL 기본값 | 255 | 1 |
| 사용 사례 | ISP 내부 | ISP 간 연결 |

**Kubernetes에서의 BGP**
MetalLB가 BGP를 활용하여 온프레미스 환경에서 LoadBalancer 타입 Service에 외부 IP를 할당. Plask 프로젝트에서 직접 구성함.

### 4. 온프레미스 인프라 전체 구조 개요

**3계층 네트워크 구조**

```
[인터넷]
   ↓
[방화벽 / VPN 게이트웨이]
   ↓
[Core Switch / L3 Switch]
   ↓
[Access Switch / L2 Switch]
   ↓
[서버 / VM / 컨테이너]
```

**주요 구성 요소 정리**

| 계층 | 구성 요소 | 역할 |
| --- | --- | --- |
| 네트워크 | Router, L3 Switch | 라우팅, VLAN 간 통신 |
| 보안 | Firewall, ACL | 트래픽 필터링 |
| 가용성 | HSRP, STP, EtherChannel | 이중화, 루프 방지 |
| 서비스 | DNS, VPN, NTP | 인프라 공통 서비스 |
| 모니터링 | Zabbix, ELK | 장애 감지, 로그 수집 |
