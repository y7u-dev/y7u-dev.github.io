# 데이터베이스

**학습 기간:** 2026.01.30 ~ 2026.02.09

---

### 1. 데이터베이스란 무엇인가

**정의**
여러 사용자와 프로그램이 동시에 접근해도 데이터의 정확성·일관성·안정성을 유지하면서 빠르게 저장·검색·수정할 수 있도록 설계된 **데이터 관리 시스템**. 데이터를 담는 그릇이 아니라 **데이터를 운영하는 엔진**이다.

**파일 vs 데이터베이스**

| 구분 | 파일 (CSV/JSON) | 데이터베이스 |
| --- | --- | --- |
| 목적 | 저장 | 서비스 운영 |
| 동시 접근 | 불가 | 가능 |
| 무결성 보장 | 없음 | 있음 |
| 장애 복구 | 없음 | 있음 |
| 성능 | 느림 | 빠름 |

**DB가 보장하는 4가지**

| 특성 | 설명 |
| --- | --- |
| 동시성 (Concurrency) | 여러 사용자가 동시 접근해도 충돌 없이 처리 |
| 무결성 (Integrity) | 결제 금액 음수 불가, 존재하지 않는 사용자 주문 불가 등 규칙 강제 |
| 지속성 (Durability) | 트랜잭션 완료 후 서버가 꺼져도 데이터 유지 |
| 성능 (Performance) | 인덱스·캐시·실행 계획으로 대용량에서도 빠른 검색 |

> "서버는 소모품이지만 데이터는 회사의 자산이다."
>

### 2. 데이터베이스 종류

**① 관계형 DB (RDBMS)**

- 행(Row)·열(Column) 기반 정형 데이터, 테이블 간 관계 정의
- SQL 사용, 스키마 고정, ACID 트랜잭션 보장, JOIN 가능
- 대표: MySQL, PostgreSQL, Oracle, MariaDB
- 사용 사례: 금융, 결제, 주문, 회원 관리

**② NoSQL (비관계형 DB)**

- 확장성·유연성 중심, Schema-less, 수평 확장(Scale-out) 용이
- ACID 대신 **BASE** 모델 채택

| 유형 | 대표 DB | 사용 사례 |
| --- | --- | --- |
| Key-Value | Redis, DynamoDB | 캐시, 세션, 토큰 |
| Document | MongoDB | 로그, 사용자 프로필, API 서버 |
| Wide Column | Cassandra, HBase | IoT, 타임시리즈, 로그 분석 |
| Graph | Neo4j | SNS 친구 추천, 관계 분석 |

**③ 인메모리 DB**

- 디스크가 아닌 RAM 기반, 매우 빠름
- 대표: Redis, Memcached
- 캐시 계층, DB 부하 감소 용도

**④ 클라우드 관리형 DB (DBaaS)**

- 설치·백업·패치·확장 자동화
- 대표: Amazon RDS, Aurora, Google Cloud Spanner

**DB 선택 기준**

| 기준 | 선택 방향 |
| --- | --- |
| 트랜잭션 중요 | RDB |
| 스키마 유연 | Document DB |
| 초고속 조회 | Key-Value |
| 대규모 분산 | NoSQL |
| 관계 탐색 | Graph DB |
| 운영 최소화 | Managed DB |

### 3. RDB (관계형 데이터베이스)

**핵심 개념**

- **ACID**: Atomicity(원자성), Consistency(일관성), Isolation(격리성), Durability(지속성)
- **정규화**: 데이터 중복을 줄이고 무결성을 높이기 위해 테이블을 분리
- **JOIN**: 여러 테이블을 관계 기반으로 결합하여 조회

**실습 내용**

- MySQL 설치 및 기본 쿼리 (SELECT, INSERT, UPDATE, DELETE)
- PostgreSQL 설치 및 JSONB 실습
- MySQL / PostgreSQL 이중화 (Primary-Replica 구성)
- 외부 접속 설정

---

### 4. NoSQL 개요

**RDB의 한계 → NoSQL 등장 배경**

| 문제 | 설명 |
| --- | --- |
| 수평 확장 어려움 | Scale-up 중심, 서버 추가 어려움 |
| 스키마 변경 부담 | 운영 중 ALTER TABLE 비용 큼 |
| 대용량 로그/이벤트 | 조인 구조에 부적합 |
| 글로벌 서비스 | 단일 DB 병목 발생 |

**RDB vs Document DB 비교 (MongoDB 기준)**

| RDB | MongoDB |
| --- | --- |
| Database | Database |
| Table | Collection |
| Row | Document |
| Column | Field |
| 스키마 고정 | 스키마 유연 |

### 5. MongoDB

**특징**

- JSON(BSON) 문서 구조로 저장
- 스키마를 DB가 강제하지 않음 → 애플리케이션이 책임
- 중첩 Document, 배열 지원
- 수평 확장에 최적화

**적합한 사례:** 로그 저장, 이벤트 수집, 사용자 프로필, AI 학습 데이터, MSA 간 데이터 교환

**부적합한 사례:** 복잡한 JOIN 필수, 금융 트랜잭션, 강한 정합성 요구

### 6. Redis

**정의**
메모리(RAM) 기반의 Key-Value 데이터 저장소. 단순 캐시가 아니라 **여러 자료구조를 제공하는 보조 데이터 저장소**.

> "Redis는 데이터를 저장하기 위한 서버가 아니라 서비스를 빠르게 만들기 위한 서버다."
> 

**지원 자료구조**

| 자료구조 | 용도 |
| --- | --- |
| STRING | 세션, 캐시 |
| LIST | 큐, 작업 대기열 |
| SET | 중복 제거, 이벤트 참여 여부 |
| HASH | 장바구니, 프로필 |
| ZSET | 랭킹, 점수 |

**TTL (Time To Live)**
데이터에 유효 시간을 설정해 자동 삭제. 로그인 세션 30분, OTP 3분, 캐시 1분 등에 활용.

**Redis vs RDB**

| 항목 | Redis | RDB |
| --- | --- | --- |
| 저장 위치 | 메모리 | 디스크 |
| 영구성 | 선택적 | 기본 |
| 목적 | 속도 | 정확성 |

**영속화 방식**

- RDB: 스냅샷 방식 (특정 시점 저장)
- AOF: 명령 로그 방식 (모든 변경 기록)

### 7. DynamoDB

**정의**
AWS에서 제공하는 완전관리형 NoSQL Key-Value & Document DB. 서버 관리 없이 자동 확장·고가용성·고성능 제공.

**핵심 개념**

| 개념 | 설명 |
| --- | --- |
| Partition Key | 데이터 분산 저장 기준, 필수 |
| Sort Key | 파티션 내 정렬 기준, 선택 |
| GSI | 다른 Partition/Sort Key로 조회 |
| LSI | 같은 Partition Key, 다른 Sort Key |
| TTL | 데이터 자동 만료 |

**읽기 모드**

- Provisioned: RCU/WCU 사전 설정 (예측 가능한 트래픽)
- On-Demand: 요청량에 따라 자동 확장 (불규칙한 트래픽)

**읽기 일관성**

- Eventually Consistent: 기본값, 빠르고 저렴
- Strongly Consistent: 최신 데이터 보장, 비용 높음

**DynamoDB vs RDB**

| 항목 | DynamoDB | RDB |
| --- | --- | --- |
| 스키마 | 유연 | 고정 |
| 확장성 | 매우 우수 | 제한적 |
| JOIN | 불가 | 가능 |
| 트랜잭션 | 제한적 | 강력 |

> DynamoDB는 **키 설계가 핵심** — 키 설계 → 성능 → 비용에 직접 영향
>
